# Platform and data architecture

## Date
2026-04-13

## Author

## Context
Our pharmaceutical data platform will need to be housed somewhere. This choice matters as it will make all downstream data engineering choices simpler or more difficult. 
For this project, I assume that we are building a data platform for a small biotech that is interested in understanding and curing one disease, but it crucially has the ambition to scale both its data operations to incorporate a greater variety of sources and its size.

The two most popular platforms for pharmaceutical data are Databricks and Snowflake. This whitepaper compares them and suggests that (quote):
-  Databricks’ Lakehouse unifies data engineering, data science, and ML on massive multi-modal datasets, making it well-suited for big-data R&D tasks like genomics, imaging, and AI model training. 
- Snowflake’s Cloud Data Platform emphasizes high-performance SQL analytics, data sharing, and ease of use for structured data workloads, with built-in governance and security to handle sensitive patient and pharma data . In practice, leading life sciences organizations often use both: for example, Snowflake may be leveraged for scalable data warehousing and collaboration (real-world evidence, biostatistics) while Databricks powers large-scale modelling and AI experiments.
Source: https://intuitionlabs.ai/articles/databricks-vs-snowflake-life-sciences

According to this whitepaper, the 4 most popular data architectures are:
- Data warehouses: 
    - Concept: Data is ingested, transformed and loaded into a central location that serves the end users.
    - Pros: Consistent data models, great for SQL queries and known use cases
    - Cons: Less flexible (incorporating new data types or unstructured data), reliance on central IT team, not designed for raw data science or real time data
- Lakehouses:
    - Concept: Raw data is massively ingested into a data lake and a unifying metadata layer is built on top provides structured data definitions and indexes.
    - Pros: Handles a variety of scientific and clinical data, good for machine learning use cases and lowers storage costs
    - Cons: Less mature architecture requires expertise, governance is challenging, slight performance cost compared to data warehouses for conventional complex SQL queries.
- Data mesh:
    - Concept: Organizational principle. Instead of a central IT team managing everything, it works via domain-oriented decentralized data ownership. A central IT team defines interoperability standards that different business domains (Research, Clinical Development, Manufacturing, Sales) follow when they own their data products.
    - Pros: Addresses data silos, organizational scalability (domains can deliver in parallel without a central IT team), domain experts closer to data can improve quality
    - Cons: Requires strong governance to avoid chaos, not ideal for small organizations, needs investments in platform engineering
- Data fabric
    - Concept: Instead of physically uniting different data products together, data fabric weaves together data from disparate sources virtually.
    - Pros: Great for complex, hybrid environments (where some data needs to be on-prem); reduces data duplication by virtualizing access; accelerates data discovery and reuse across silos. 
    - Cons: Can be technologically complex to implement; performance might suffer if many queries go across systems (thus often a fabric is paired with a physical warehouse/lake for heavy crunching, and fabric is used for less intensive integration or discovery tasks)

Source: https://intuitionlabs.ai/articles/data-warehousing-in-life-sciences

## Decision
- [Platform] Databricks will be the platform of choice, since:
    - This use case fits more into the experimental sphere and requires unifying disparate sources, something Databricks is recognized to be best at.
    - For a biotech operating under GxP principles and working toward FAIR data standards, the platform must support full audit trails, access control, and reproducible experiment records. Databricks addresses this through:

        - Unity Catalog: centralized access control and data lineage, supporting GxP audit requirements
        - MLflow: versioned experiment tracking for reproducibility (FAIR principle: Reusable)
        - Delta Lake: ACID transactions ensuring data integrity across the ingestion pipeline
        - HIPAA and 21 CFR Part 11 compliance fall outside this project's current scope (no patient data is processed), but the architecture is designed to support them when the platform scales to clinical data.
    - A careful consideration on cost is needed, but for the use case that we have in mind (collecting disparate scientific data for ML/AI purposes) Databricks seems to be superior technically.
    - This project scopes to demonstrating Databricks capabilities on a representative biotech use case. The technical fit for this use case is well-established. A production decision would additionally require a total cost of ownership analysis, an organizational fit assessment, and a review of existing vendor contracts.
- [Platform] For this public project, we'll use Databricks Free edition.
- [Architecture] Data mesh will be the option we go for here, since:
    - Data warehouses are not suitable for the disparate data we need to bring together.
    -  Data Fabric is designed for environments with cross-organizational data ownership constraints or hybrid on-premises/cloud requirements. Neither applies here: all source data is open-access and will be physically co-located in cloud object storage. This decision should be revisited if the platform incorporates data that is contractually restricted in its physical location or sensitive enough to require virtualized access.
    - Both Lakehouses and Data Mesh are good options. I'm opting for Data Mesh, because:
        - Since the goal is to demonstrate knowledge in data management, I should design it with the challenges of big organizations in mind.
        - Contrarian take: Shifting from Lakehouses to Data Mesh requires not only technical investments but also change management. If your Research department never had to handle data products before, how will you convince them to take up that expanded role down the line? I think the author underestimates the change management required and I think that starting off with proper responsibilities will make later scaling a lot smoother.

- [Architecture] The fictitious company will be organized into Research, Clinical, Omics and Manufacturing departments. This needs to map to your org chart. We will assume that their will be a data product owner in each department that knows both the business and the technical requirements. In this toy case, that person will be me.

## Status
Accepted.

## Consequences
Choosing Data Mesh over a Lakehouse shifts data quality and curation responsibility from a central IT team to domain owners. This is the primary organizational risk: if domain owners lack the tooling, time, or mandate to treat their data as a product, the mesh degrades into a collection of inconsistent silos — the very problem it is meant to solve.
Concretely, this means each domain (Research, Clinical, Omics, Manufacturing) must maintain a catalogued, versioned, and access-controlled data product in Unity Catalog. The standards that govern this — naming conventions, SLAs, access policies, lineage requirements — will be defined in a dedicated governance ADR.
